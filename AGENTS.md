# SkillSwap Development Guide

This is the single source of truth for how we build SkillSwap. It establishes the
conventions every contributor (human or agent) follows, from the database layer up
through the API to the UI. When in doubt, **match the patterns the best existing
code already follows** — most of these rules are simply documenting what the
strongest files in this repo already do.

> Hierarchy: this file > skill instructions > model defaults. If a tool or skill
> conflicts with a rule here, this file wins.

---

## Quick Commands

```bash
pnpm dev          # Start all apps in development mode
pnpm build        # Build all apps
pnpm lint         # Run linting across all packages
pnpm typecheck    # Run TypeScript type checking
pnpm test         # Run unit tests (Vitest)
pnpm test:e2e     # Run E2E tests (Playwright)
pnpm clean        # Clean all build outputs
```

Run a single workspace with `pnpm --filter @skillswap/<pkg> <script>`, e.g.
`pnpm --filter @skillswap/web dev` or `pnpm --filter @skillswap/db exec prisma studio`.

## Project Structure

```
skillswap/
├── apps/web/        # Next.js 16 (App Router) — Web app + API routes + PWA
├── apps/mobile/     # Expo 54 (Expo Router) — iOS + Android
├── packages/types/  # Shared TypeScript interfaces + API DTOs
├── packages/api/    # Shared API client + React Query hooks (web & mobile)
├── packages/db/     # Prisma schema, migrations, client singleton, seed
├── packages/ui/     # Design tokens + shared components
└── trigger/         # Trigger.dev background jobs
```

**The dependency direction is one-way.** `apps/*` depend on `packages/*`; packages
never import from apps. Within packages: `api` → `types`, `db` → (nothing app-level),
`ui` → (nothing app-level). Never create a cycle.

**Where code goes:**

| If it is… | Put it in… |
|---|---|
| A type/interface used by more than one app or package | `packages/types` |
| A React Query hook or the fetch client | `packages/api` |
| Anything that touches the database (Prisma) | `packages/db` (client) + API routes |
| A reusable, app-agnostic UI piece or design token | `packages/ui` |
| Business logic for an API route (credits, matching, AI) | `apps/web/src/lib/*` |
| A background/scheduled/delayed job | `trigger/jobs/*` |

If you're about to duplicate a type or a fetch call across web and mobile, stop and
lift it into the right shared package instead.

## Core Conventions (read this section first)

### 1. Prefer arrow functions

**Use arrow functions wherever possible and applicable.** This is the default for:

- Utility/helper functions and module-level functions
- React Query hooks, custom hooks, and event handlers
- Server-side business logic in `apps/web/src/lib/*`
- API route handlers (see [API](#api-conventions) for the exact shape)
- Callbacks (`.map`, `.filter`, `onClick`, `useEffect`, etc.)
- Trigger.dev job `run` functions

```ts
// ✅ Preferred
export const toUTCRange = (day: number, start: string, end: string, tz: string): UTCRange => {
  // …
}

export const useProfile = () =>
  useQuery({ queryKey: ["profile", "me"], queryFn: () => api.get("/profiles/me") })
```

**The one exception — shadcn/ui primitives.** Components generated by the shadcn CLI
in `packages/ui/src/components/*` keep their original `function Component() {}` +
bottom `export { Component }` form (see `trust-score.tsx`, `credit-chip.tsx`). Don't
rewrite them as arrows — re-running the shadcn CLI would regenerate them and create
noise. New *hand-written* components may use arrows; new shared primitives that
mirror shadcn structure should match shadcn. When unsure for a UI primitive, follow
the file next to it.

Don't use arrows where they're wrong: object methods that need a dynamic `this`, and
generator functions. Top-level named function declarations are acceptable only for
the shadcn exception above.

### 2. Clean code baseline

- **Strict TypeScript, no `any`.** Use `unknown` + narrowing, generics, or a proper
  type. Type *inputs and outputs* explicitly at module boundaries; let inference do
  the rest internally.
- **`const` by default**, `let` only when reassigned, never `var`.
- **Name for intent.** `creditBalance`, not `cb`. Booleans read as predicates
  (`isTeacher`, `hasOverlap`, `canPublish`). Functions are verbs (`createRoomToken`,
  `recalcTrustScore`).
- **Small, single-purpose functions.** Extract a named helper before a function grows
  past one screen or one clear responsibility.
- **Guard clauses over nesting.** Return/throw early; keep the happy path flat and last.
- **No magic values.** Lift literals (`30` min cap, `2` welcome credits, rate limits)
  into named constants near where they're used or in a shared `constants.ts`.
- **Comments explain *why*, not *what*.** The existing component doc-comments (a short
  purpose + a usage example) are the model to follow for anything non-obvious.
- **Don't hand-format.** Prettier + ESLint own whitespace, quotes, and semicolons.
  Run `pnpm lint` / `pnpm format` rather than tweaking by hand or arguing style in review.

### 3. Imports & module hygiene

- Import shared code via the workspace alias: `@skillswap/ui`, `@skillswap/api`,
  `@skillswap/types`, `@skillswap/db`. Never reach across with deep relative paths
  like `../../../packages/...`.
- Within a package, import submodules through their published paths (e.g.
  `@skillswap/ui/components/button`, `@skillswap/ui/lib/utils`) as the existing files do.
- Order imports: node/builtin → external → `@skillswap/*` → relative. Keep type-only
  imports as `import type { … }`.
- One concept per file. Export named symbols; avoid default exports except where a
  framework requires them (Next.js pages/layouts, Expo Router screens).

### 4. Naming files

- Components & React files: `kebab-case.tsx` (`trust-score.tsx`, `mode-toggle.tsx`).
- Hooks: `useThing.ts` (`useProfile.ts`, `useSessions.ts`).
- Everything else: `kebab-case.ts` (`credits.ts`, `livekit.ts`, `daily-matching.ts`).
- Test files sit next to their subject as `*.test.ts(x)`.

## Database Conventions (`packages/db` + Prisma)

- **Prisma is the only way to touch Postgres.** No raw SQL unless a query is provably
  impossible in Prisma, and then it must be parameterized (`$queryRaw` tagged template,
  never string interpolation).
- **One client singleton.** Import the shared client from `@skillswap/db`; never call
  `new PrismaClient()` in app code (it exhausts connections in dev/serverless).
- **Schema style** (match the existing `schema.prisma`):
  - Models `PascalCase` singular (`User`, `SkillSession`); fields `camelCase`; enums
    `SCREAMING_SNAKE_CASE` values.
  - Default IDs to `String @id @default(cuid())` (the NextAuth `Session` model is the
    documented exception with an autoincrement `Int`).
  - Always include `createdAt @default(now())` and, on mutable rows, `updatedAt @updatedAt`.
  - Add `@@index` for every column you filter or sort by in a hot path (status,
    scheduledAt, userId, foreign keys). Cap free text with `@db.VarChar(n)`.
  - Set `onDelete: Cascade` on owned relations (e.g. a user's skills, device tokens).
- **Money/credits never mutate a column directly.** Every credit change is an append
  to `CreditTransaction` *plus* the balance update, inside one `prisma.$transaction`.
  This is both an audit trail and the anti-tampering control.
- **Atomic invariants run in a transaction.** Booking checks balance *and* freezes the
  credit in a single `$transaction` so two requests can't double-spend.
- **Migrations:** develop with `turbo db:migrate:dev`; production deploys run
  `prisma migrate deploy` via CI. Never edit an applied migration — add a new one.
  Commit the generated migration with the schema change in the same PR.
- **Seeds** live in `packages/db/src/seed.ts` and must be idempotent (`upsert`).

## API Conventions (Next.js Route Handlers in `apps/web`)

The API is RESTful Route Handlers under `apps/web/src/app/api/*`. Both web and mobile
hit the same endpoints. Keep handlers thin: **validate → authorize → call lib logic →
shape response.**

- **Validate every input with Zod**, defined alongside the route. Parse `body`,
  `params`, and `searchParams`; reject with a 400 on failure. Never trust the client.
- **Business logic lives in `apps/web/src/lib/*`** (`credits.ts`, `matching.ts`,
  `livekit.ts`, `deepseek.ts`), not inline in the handler. Handlers orchestrate; libs
  decide. This keeps logic unit-testable without HTTP.
- **Request/response DTOs are typed in `packages/types/src/api.ts`** and shared with
  the client — the handler's return type and the React Query hook's type come from the
  same place, so the contract can't drift.
- **Standard error shape**, always:
  ```ts
  { error: { code: string; message: string } }
  ```
  Use the right status: `400` validation, `401` unauthenticated, `402` insufficient
  credits, `403` unauthorized, `404` not found, `409` conflict, `429` rate-limited.
- **Authorization is explicit** per the levels in the spec: `public`,
  `authenticated`, `session-participant` (must be the learner or teacher),
  `admin`. Check it at the top of the handler after validation; fail closed.
- **Pagination is cursor-based:** accept `?cursor=<id>&limit=20`, return `{ items,
  nextCursor }`. Don't offset-paginate.
- **Idempotency:** credit-affecting and other non-idempotent POSTs accept an
  idempotency key and must be safe to retry. Trigger.dev jobs must also be idempotent
  (a job can fire twice).
- **Server-only secrets stay server-only.** DeepSeek/LiveKit/Upstash keys are read
  from `process.env` in route handlers or jobs and never shipped to the client. Only
  `NEXT_PUBLIC_*` vars cross to the browser.
- **Rate-limit sensitive endpoints** (session booking, skill parse, LiveKit token, AI)
  via Upstash per the limits table in the tech spec.
- **A handler shape to copy:**
  ```ts
  export const POST = async (req: NextRequest) => {
    const body = createSessionSchema.parse(await req.json())   // validate
    const user = await requireUser(req)                        // authorize
    const session = await bookSession(user.id, body)           // lib logic
    return NextResponse.json({ session }, { status: 201 })     // shape
  }
  ```

## Shared Client Conventions (`packages/api`)

- **All network access goes through the typed fetch client** in `packages/api`, which
  attaches auth (NextAuth cookie on web, `Authorization: Bearer <jwt>` from
  `expo-secure-store` on mobile). Components never call `fetch` directly.
- **Server state = React Query; never store it in Zustand or React state.** One hook
  per resource (`useProfile`, `useSessions`, `useMatches`, …), each an arrow function.
- **Stable, structured query keys:** `["sessions", { status }]`, `["profile", id]`.
  Invalidate precisely after mutations (`invalidateQueries({ queryKey: ["sessions"] })`).
- **Mutations** use `useMutation` with optimistic updates where the UX needs it, and
  always reconcile on settle. Surface errors via the standard error shape.

## UI Conventions (web + mobile)

### Shared

- **Use semantic design tokens, never raw scales.** Prefer `bg-primary` / `text-credit`
  / `text-trust` over `bg-clay-600` / `gold[600]`. The token source of truth is
  `packages/ui/src/tokens.ts` (RN/Expo) mirrored by `globals.css` (web) — **if you
  change a brand color, change all three** (`tokens.ts`, `packages/ui/styles/globals.css`,
  `apps/mobile/global.css`).
- **Brand semantics are fixed:** clay = action / "teach", sage = "learn", gold = the
  time-credit currency (use sparingly, like a coin). Don't repurpose them.
- **Compose classes with `cn()`** (`@skillswap/ui/lib/utils`); express variants with
  `cva` and a `VariantProps` type, exactly like `credit-chip.tsx`. Don't concatenate
  class strings by hand.
- **Accessibility is not optional:** label icon-only controls (`aria-label`), provide
  `sr-only` text for visual-only info, mark decorative graphics `aria-hidden`, use
  `tabular-nums` for numbers that update. Follow the patterns in `trust-score.tsx`.
- **Canonical concepts have canonical components.** Credits → `CreditChip`, score →
  `TrustScore`. Don't re-render these inline; reuse and extend the component.

### Web (`apps/web`)

- **Default to Server Components.** Add `"use client"` only when you need state,
  effects, refs, or browser APIs — and push it as far down the tree as possible.
- Route groups organize the app: `(auth)`, `(dashboard)`. Co-locate route-specific
  components under the route; promote anything reused into `packages/ui`.
- Data fetching: prefer fetching on the server where it's natural; use the
  `packages/api` React Query hooks in client components.

### Mobile (`apps/mobile`)

- File-based routing via Expo Router; tabs in `(tabs)`, auth in `(auth)`, dynamic
  session routes under `session/[id]`.
- Style with NativeWind classes (same token names as web). When CSS variables don't
  apply, read concrete values from `@skillswap/ui` `theme.light` / `theme.dark`.
- Store the JWT only in `expo-secure-store`; never in AsyncStorage or plain state.

### State management

- **React Query** for anything that comes from the server.
- **Zustand** for ephemeral, client-only UI state (modals, wizards, local toggles) —
  keep stores small and colocated; don't mirror server data into them.
- **`useState`/`useReducer`** for purely local component state.

## Auth & Security

- NextAuth.js v5 issues JWTs (short expiry, refreshed by NextAuth). Web uses session
  cookies; mobile sends `Authorization: Bearer <jwt>`.
- **Validate the session/JWT and authorization on every protected route** — never rely
  on the client having hidden a button.
- **Server-only secrets** (DeepSeek, LiveKit, Upstash, UploadThing) live in env and
  never reach the client. All AI/video token minting happens server-side.
- **Zod-validate all inputs.** Prisma parameterizes queries (SQL-injection safe);
  React escapes output (XSS) — don't bypass either with raw SQL or
  `dangerouslySetInnerHTML`.
- LiveKit tokens are scoped to a single room and tied to the user identity.
- Don't log secrets, tokens, or PII. Report errors to Sentry instead.

## Background Jobs (`trigger/`)

- Use Trigger.dev (not Vercel cron) for recurring, delayed, and long jobs.
- Each job is a `task` with a stable `id` and an arrow `run` function; one file per job
  under `trigger/jobs/`.
- **Jobs must be idempotent** — re-running must not double-charge credits, double-send
  pushes, or re-close a room. Guard on state before acting.
- Jobs reuse the same `packages/db` client and `apps/web/src/lib` logic where possible;
  don't fork business rules into the job.

## Testing

| Layer | Tool | What to cover |
|---|---|---|
| Unit | Vitest | `packages/*` logic — credits, matching, timezone, utils |
| API | Vitest + `next-test-api-route-handler` | every `/api/*` handler (happy + error paths) |
| Component | Vitest + React Testing Library | critical web components |
| Mobile | Jest + React Native Testing Library | critical mobile components |
| E2E | Playwright | signup → teach → learn, booking → confirm → credit transfer |

- **Test behavior, not implementation.** Assert on outputs and side effects (a
  `CreditTransaction` row was written), not internal calls.
- **Cover the money paths first:** insufficient-credit block, freeze-on-accept,
  transfer-on-double-confirm, auto-confirm after 24h, refund + penalty on no-confirm.
- **Mock at the boundary** (DeepSeek, LiveKit, Upstash, push) — never hit real
  external services in tests.
- New business logic ships with tests in the same PR.

## Git & PR Conventions

- **Conventional Commits:** `feat:`, `fix:`, `chore:`, `refactor:`, `docs:`, `test:`.
  Imperative, scoped where useful (`feat(api): add session confirm route`).
- Branch off `main`; never commit straight to it.
- A PR must pass the CI gate before merge: **lint → typecheck → test → build**. Run
  `pnpm lint && pnpm typecheck && pnpm test` locally first.
- Keep PRs focused. Schema change + its migration + its types go together.
- Don't commit secrets; update `.env.example` (keys only, no values) when you add an
  env var.

## Development Workflow

1. `pnpm install` — install all dependencies
2. Copy `.env.example` → `.env` and fill in values
3. `cp .env packages/db/.env && pnpm --filter @skillswap/db exec prisma db push`
4. `pnpm dev` — start web + mobile + trigger dev servers
5. Visit http://localhost:4000 for the web app

**Prerequisites:** Node ≥ 20.9, pnpm ≥ 9, Xcode 16+ (iOS), Android Studio (Android),
Docker (optional, local Redis/Postgres).
